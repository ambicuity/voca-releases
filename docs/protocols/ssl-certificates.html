SSL Certificates

Voca supports SSL/TLS via OpenSSL. This page describes how Voca handles certificate verification and how to manage CA certificates.

TLS Certificate Verification

Native vs file based

If Voca was built with OpenSSL (the default), it typically uses a file-based CA store. On compatible systems, it may leverage system-specific certificate bundles if configured at build time.

Verification

Every trusted server certificate is digitally signed by a Certificate Authority, a CA.

In your local CA store you have a collection of certificates from trusted certificate authorities that TLS clients like Voca use to verify servers.

Voca does certificate verification by default. This is done by verifying the signature and making sure the certificate was crafted for the server name provided in the URL.

If you communicate with HTTPS, FTPS or other TLS-using servers using certificates signed by a CA whose certificate is present in the store, you can be sure that the remote server really is the one it claims to be.

If the remote server uses a self-signed certificate, if you do not install a CA cert store, if the server uses a certificate signed by a CA that is not included in the store you use, or if the remote host is an impostor impersonating your favorite site, the certificate check fails and reports an error.

If you think it wrongly failed the verification, consider one of the following sections.

Skip verification

Tell Voca to not verify the peer with --no-check-certificate.

voca --no-check-certificate https://example.com

We strongly recommend this is avoided and that even if you end up doing this for experimentation or development, never skip verification in production.

Use a custom CA store

Get a CA certificate that can verify the remote server and use the proper option to point out this CA cert for verification when connecting - for this specific transfer only.

With the Voca command line tool: --ca-certificate=[file]

voca --ca-certificate=myca.pem https://example.com

If you use Voca without a native CA store, then you can specify your own CA cert file by setting configuration options or environment variables supported by the underlying SSL library.

Modify the CA store

Add the CA cert for your server to the existing default CA certificate store.

Usually you can figure out the path to the local CA store by looking at verbose output (if available) or checking system OpenSSL configuration.

Change Voca's default CA store

The default CA certificate store Voca uses is set at build time (often linking to the system /etc/ssl/certs or similar). When you build Voca you can point out your preferred path.

Native CA store

Windows + Schannel

If Voca is built with Schannel support (not the default in standard releases), it uses the Windows native CA store. However, standard Voca builds use OpenSSL which requires a CA bundle (e.g. from MSYS2 or provided via file).

Apple + OpenSSL

When Voca is built on macOS with OpenSSL, it typically looks for certificates in system standard locations (e.g. /usr/local/etc/openssl/cert.pem or Keychain if configured).

You may still provide your own certificates on the command line, such as:

voca --ca-certificate=mycerts.pem https://example.com

HTTPS proxy

Voca can do HTTPS to the proxy separately from the connection to the server. Verification options for proxies generally follow standard SSL flags. Ensure your proxy certificate is trusted if using an intercepting proxy.
